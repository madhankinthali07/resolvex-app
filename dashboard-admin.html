AI Engine v3.2 (Hybrid Architecture)

Deep Hybrid AI Analysis & Decision Logic

ResolveX integrates a sophisticated hybrid architecture combining a deterministic Business Rules Engine (4 rule types, 22+ individual rules) for immediate policy, safety, and sync-failure overrides with a multi-layered Machine Learning Pipeline for statistical solvability fallback across all 16 ticket subjects.

Hybrid AI Decision Pipeline Flow

1. Ticket Ingest
Ticket Metadata & Description
2. Contradiction
System Record vs Reality Check
3. Severity Rules
Hacking/Safety Override Check
4. Day SLA Rules
Decisive Policy Check (Yes/No)
5. Attempt Rules
Decisive Retry Check (Yes/No)
6. ML Classifier
TF-IDF NLP + Model Fallback

Deterministic Rules Engine

Contradiction Rules — 13 Rules (Priority Override) Scans for backend record vs. reality discrepancies across all 13 subject areas: Refund marked done but not received, Delivery marked done but not received, Cancellation confirmed but still charged, Fix marked resolved but issue persists, Hardware marked repaired but still faulty, Battery marked fixed but still draining, Peripheral marked compatible but won't pair, Software marked compatible but won't sync, Account marked restored but still locked, Data marked recovered but still missing, Installation marked complete but won't open, Setup marked complete but device not working, Display marked fixed but visual issue persists, Recommendation doesn't match stated need. Escalates to a human immediately regardless of time elapsed — even on day 1 — since these indicate a backend processing failure, not normal delays.
Severity Rules — 4 Rules (Instant Escalation) Instantly escalates to a human without any time/attempt math for four critical risk classes: (1) Security / Account Compromise — hacked, unauthorized access, fraud, breach, identity theft; (2) Irreversible Data Loss — lost all data, permanently deleted, no backup; (3) Billing Dispute — double charge, unauthorized charge, overcharged; (4) Physical Safety Hazard — smoke, burning smell, caught fire, sparking, severe overheating, explosion risk. Safety hazard tickets are never routed to AI since the risk to property/person is too high.
Day SLA Rules — 4 Policies (Decisive Timelines) Extracts elapsed-day figures from free text (digit patterns, word numbers like "two weeks", calendar dates) and checks against four policy windows: Refund SLA (30 days — includes billing inquiries), Cancellation SLA (14 days), Delivery SLA (10 days), Hardware Warranty (365 days — covers broken/defective/stopped-working hardware), Software Update Grace Period (60 days — covers bugs persisting since an update). If elapsed days exceed the window → escalate to human. If within the window → decisive "Yes, AI can handle" (normal processing time). If no day count is found in the description despite the topic matching → falls through to ML.
Attempt Rules — 1 Rule, 10 Subjects (Failed Self-Fixes) Counts how many times the customer says they already tried fixing it themselves ("tried 3 times", "reinstalled twice", "restarted multiple times"). A bare single mention with no count is treated as 1 attempt. Applies across 10 subjects: Software Bug, Hardware Issue, Network Problem, Account Access, Installation Support, Product Setup, Peripheral Compatibility, Display Issue, Data Loss, Battery Life. Threshold: > 2 attempts → escalate to human (repeating automated steps that already failed is unproductive). ≤ 2 attempts → decisive "Yes" (AI guides them through standard troubleshooting first).

Machine Learning Classifier Fallback

When ML Activates ML only runs when no business rule's topic applies at all — primarily for Product Compatibility and Product Recommendation tickets (no objective deadline or retry threshold exists for open-ended compatibility/recommendation questions), and for any topic where a day/attempt rule topic matched but no count could be extracted from the description.
Natural Language Processing (NLP) + Feature Encoding Converts free-text ticket descriptions to numerical features using a TF-IDF Vectorizer (max 1,000 features, English stop-word removal). Combines with One-Hot Encoding of structured fields (Product Purchased, Ticket Type, Subject, Priority, Contact Channel) into a single feature matrix.
Best Pipeline Selection (F1-Score Tournament) Trains and benchmarks three models — Logistic Regression, Random Forest, and XGBoost (if available) — on an 80/20 train-validation split. Automatically selects the highest F1-Score winner and re-fits it on the complete labeled dataset. Final prediction uses a 0.5 solvability probability threshold: ≥ 0.5 → AI can resolve; < 0.5 → route to human agent.
Solvability Label Construction Since the dataset has no direct "AI solvable" label, a composite satisfaction score is computed per ticket using priority weights, channel weights, ticket-type weights, and keyword sentiment (positive: "solved", "resolved", "helpful"; negative: "broken", "crash", "error", "refund"). Tickets scoring ≥ 0 are labeled Satisfied (AI-resolvable). A small noise term (σ=0.1) prevents overfitting to edge cases.
